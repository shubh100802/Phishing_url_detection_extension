RUDRAKSHA: PHISHING DETECTION SYSTEM - ENHANCED PROJECT OVERVIEW
================================================================================

PROJECT OVERVIEW
===============
Rudraksha is a comprehensive, full-stack phishing detection system designed to provide real-time URL scanning, threat management, and user-friendly dashboards. Built with modern web technologies, it offers role-based authentication, analytics, and extensible architecture for future enhancements like machine learning integration and browser extensions.

**Key Features:**
- **Real-time URL Scanning:** Heuristic-based risk assessment with configurable depth (quick/standard/deep).
- **Role-Based Access Control:** Separate dashboards for admins (threat management, user oversight) and users (personal scanning, history).
- **Threat Intelligence:** Automated threat detection, status tracking, and management.
- **Responsive UI:** Modern, animated interfaces using vanilla HTML/CSS/JS.
- **API-Driven Architecture:** RESTful endpoints for scanning, analytics, and management.
- **Security-First Design:** JWT authentication, bcrypt hashing, rate limiting, and CORS protection.
- **Extensible Framework:** Ready for Chrome extension and ML microservice integration.

**Business Value:**
- Protects users from phishing attacks through proactive scanning.
- Provides admins with tools for threat monitoring and user management.
- Scalable for enterprise use with MongoDB persistence.
- Open-source foundation for community contributions.

ARCHITECTURE OVERVIEW
=====================
The system employs a modular, layered architecture separating concerns for maintainability and scalability.

**Core Components:**
1. **Backend (Node.js/Express):** API server handling authentication, scanning, and data management.
2. **Frontend (Vanilla Web):** Responsive dashboards and authentication pages.
3. **Database (MongoDB):** NoSQL storage for users, scans, and threats.
4. **Extension (Chrome):** Browser integration for on-the-fly scanning (planned).

**Technology Stack:**
- **Backend:** Node.js 18+, Express.js 4.x, MongoDB 6.x, Mongoose ODM.
- **Frontend:** HTML5, CSS3 (Grid/Flexbox), Vanilla JavaScript (ES6+).
- **Security:** Helmet, CORS, express-rate-limit, JWT (jsonwebtoken), bcryptjs.
- **Development Tools:** Nodemon, dotenv, Morgan (logging).
- **Testing:** Jest/Mocha (planned), Postman for API testing.
- **Deployment:** Docker (planned), PM2 for production.

**Data Flow:**
1. User authenticates → JWT token issued.
2. Frontend stores token in localStorage.
3. API requests include Bearer token in Authorization header.
4. Scanning triggers heuristic analysis → Results stored in DB.
5. Threats auto-generated for risky URLs.
6. Admins manage via dedicated endpoints.

BACKEND DETAILS (ENHANCED)
===============
**Server Configuration (server.js):**
- Express app with JSON/URL-encoded parsing and cookie support.
- Security: Helmet (headers), CORS (localhost origins), rate limiting (configurable).
- Logging: Morgan in dev mode, Winston in production (planned).
- Database: MongoDB connection with connection pooling and error handling.
- Admin Seeding: Conditional seeding based on env flag and DB state.
- Error Handling: Global middleware for 404s and errors.
- Startup: Graceful server start with port fallback.

**Database Models (Detailed):**
1. **User Model:**
   - Schema: name (String, required), email (unique), password (hashed), role (enum: admin/user), status (enum: active/suspended), timestamps.
   - Methods: comparePassword(), toJSON() (excludes password).
   - Hooks: Pre-save password hashing.
   - Indexes: email (unique), role.

2. **Scan Model:**
   - Schema: userId (ObjectId ref), url (String), verdict (enum), riskScore (Number 0-1), sources (Array), options (Object), timestamp.
   - Indexes: compound (url, createdAt), userId.

3. **Threat Model:**
   - Schema: url (unique), level (enum: low/medium/high), status (enum: blocked/investigating/monitored), notes (String), detectedAt.
   - Indexes: url, status.

**Authentication System:**
- JWT: Stateless tokens with configurable expiration (default 7d).
- Passwords: bcrypt with 12 rounds for security.
- Middleware: authenticate() (verifies token), authorize() (checks roles).
- Seeding: Creates admin user on first DB connect if enabled.

**API Controllers (Expanded):**
1. **Auth Controller:**
   - Register: Validates input, checks uniqueness, hashes password.
   - Login: Supports email/username, verifies credentials, returns JWT + user data.

2. **Scan Controller:**
   - Scan: Accepts URL + options, runs heuristics, returns verdict/risk/sources.
   - Heuristics: Keywords (login, bank), TLDs (.ru, .tk), IP patterns, hyphens, length, subdomains.
   - Scoring: Weighted algorithm, thresholds for verdicts.

3. **Threats Controller (Admin):**
   - List: Paginated with filters (status, level).
   - Update: PATCH for status/level/notes.

4. **Users Controller (Admin):**
   - List: Paginated users.
   - Create/Update: Role/status management.

5. **Analytics Controller (Admin):**
   - Stats: Aggregated counts, uptime, recent activity.

**API Endpoints Summary:**
- Auth: POST /api/auth/register, POST /api/auth/login
- Scan: POST /api/scan
- Threats: GET /api/threats, PATCH /api/threats/:id
- Users: GET /api/users, POST /api/users, PATCH /api/users/:id
- Analytics: GET /api/analytics
- Health: GET /api/health

**Configuration:**
- Env Vars: PORT, MONGODB_URI, JWT_SECRET, CORS_ORIGINS, RATE_LIMITS, etc.
- Validation: Joi or similar for env parsing (planned).

KEY TECHNICAL DETAILS (ENHANCED)
=====================
**Security Features:**
- Headers: Helmet for XSS/CSRF protection.
- Authentication: JWT with short expiration, refresh tokens (planned).
- Encryption: bcrypt for passwords, HTTPS enforcement (planned).
- Validation: Input sanitization, SQL injection prevention via Mongoose.
- Monitoring: Rate limiting, logging, audit trails.

**Scanning Algorithm (Detailed):**
- Input: URL string.
- Parsing: Extract domain, TLD, subdomains.
- Heuristics: Keyword matching (e.g., 'paypal', 'login'), suspicious patterns.
- Scoring: Additive weights, normalized to 0-1.
- Verdicts: Safe (<0.45), Suspicious (0.45-0.75), Malicious (>0.75).
- Deep Mode: Additional checks (e.g., external API calls planned).

**Performance Considerations:**
- Database: Indexing for fast queries, connection pooling.
- API: Caching (Redis planned), pagination for large datasets.
- Frontend: Lazy loading, minification (planned).
- Scalability: Horizontal scaling with load balancers (planned).

**Testing Strategy:**
- Unit: Model methods, utility functions.
- Integration: API endpoints, auth flows.
- E2E: User journeys via Puppeteer.
- Coverage: Aim for 80%+ with Istanbul.

**Deployment Guide:**
- Local: VS Code Live Server for frontend, npm run dev for backend.
- Production: Docker containerization, PM2 clustering.
- Cloud: Heroku/AWS with MongoDB Atlas.
- CI/CD: GitHub Actions for automated testing/deployment.
